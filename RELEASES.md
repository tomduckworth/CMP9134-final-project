All old endpoints work perfectly.Next Version: v1.3.0Justification: This change introduces a new, backward-compatible feature to the public API. Existing client integrations will continue to function without any modifications. Under the SemVer specification, adding public functionality in a backward-compatible manner requires incrementing the MINOR version (the second digit) and resetting the PATCH version to 0.Scenario C: Redesigned JSON PayloadScenario description: You completely redesign the JSON payload required for the /api/move endpoint. If old frontends try to send the old payload, they will receive a 422 Validation Error.Next Version: v2.0.0Justification: This change is a backward-incompatible, breaking API change. Older clients or frontends will break immediately upon deployment because they are transmitting an obsolete payload structure. According to the SemVer specification, any breaking change that disrupts backward compatibility requires incrementing the MAJOR version (the first digit) and resetting both the MINOR and PATCH versions to 0.
