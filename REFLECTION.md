Academic Critical Reflection (Borton's Model)
This document provides a critical reflection on a key engineering challenge tackled during the development of the Ground Control Station API, mapping technical decisions to Legal, Ethical, Professional, and Social Issues (LEPSI).
Paragraph 1: What? (Objective Description)
During the integration of the legacy mission statistics module in Workshop 12, I inherited significant technical debt and critical security vulnerabilities. The original calc_stats endpoint had high cyclomatic complexity (heavily nesting conditional structures), suffered from an extreme SQL injection vulnerability due to direct, unparameterized Python f-string database writes, and relied on manual dictionary key checks that bypassed FastAPI's automated validation engine. My task was to safely refactor this unstable module, introducing strong data validation via Pydantic schemas, restructuring control flow with early-return guard clauses, sanitizing query logs, and building a continuous integration pipeline with automated flake8 linting and pytest coverage testing.
Paragraph 2: So What? (LEPSI Implications)
This refactoring was not merely an aesthetic optimization, but a fundamental requirement under professional, ethical, and legal engineering frameworks. From an ethical and professional standpoint, leaving an unparameterized SQL database command in place violates Section 2 of the British Computer Society (BCS) Code of Conduct, which mandates a strict duty of professional competence and a requirement to prioritize safety and cybersecurity. Legally, exposing a system to SQL injection compromises data integrity, which directly breaches the security requirements of Article 32 of the General Data Protection Regulation (GDPR). Furthermore, failing to construct automated testing to verify system limits poses physical dangers in cyber-physical automation; a robot moving out of boundaries due to unvalidated, overflowed calculation scores can lead to catastrophic hardware damage or safety hazards in shared human workspaces.
Paragraph 3: Now What? (Technical Resolution & Future Learning)
To address these concerns, I isolated the business logic and successfully refactored the module to adhere to industry standards. I introduced the MissionStatsRequest Pydantic model to automate input verification, upgrading validation failures to formal HTTP 422 Unprocessable Entity responses. I flattened nested conditionals into clean guard clauses, implemented safe, parameterized query database logs to eliminate SQL injection vectors, and deployed a GitHub Actions CI pipeline running automated linter tasks. This experience has deeply reshaped my perspective as a software engineer; I now understand that technical debt is inherently a security and ethical vulnerability. In future development cycles, I will employ "Security by Design" and write test coverage before altering complex logic to ensure safety boundaries remain fully locked.
